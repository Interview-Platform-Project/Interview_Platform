# Interview Platform

Учебный проект: сервис проведения технических и мок-интервью (realtime-редактор
кода, видео, AI-подсказки, витрина участников, обратная связь, лидерборд,
уведомления в Telegram).

> Это **скелет**. Здесь намеренно нет стека и конфигов — только направление.
> Каждая команда поднимает своё приложение с нуля и наслаивает инструменты по
> мере разработки.

## Структура

```
interview-platform/
├─ apps/
│  ├─ backend/          # серверное приложение (поднимает команда бэкенда)
│  └─ frontend/         # клиентское приложение (поднимает команда фронтенда)
├─ packages/
│  └─ shared/           # общий код (типы/контракты/константы) — пока пустой
├─ package.json         # корень монорепы
└─ pnpm-workspace.yaml  # состав воркспейса (apps/*, packages/*)
```

## Монорепа

Управление пакетами — **pnpm** (см. `pnpm-workspace.yaml`). Приложения живут в
`apps/*`, переиспользуемый код — в `packages/*`.

## С чего начать команде

1. Определиться со стеком своего приложения (`apps/backend` / `apps/frontend`).
2. Инициализировать проект внутри своей папки, добавить зависимости.
3. По мере необходимости наслаивать общее: линтеры, форматтер, тесты, Docker,
   CI/CD, observability и т.д.
4. Общие типы/контракты между бэком и фронтом класть в `packages/shared`.

Договорённости команды (git-флоу, код-ревью, созвоны, доска) фиксируются
отдельно по мере старта.

## Branch name pattern

feat/ — новая функциональность / страница / компонент.

update/ — доработка или обновление функциональности / страницы / компонента.

fix/ — исправление ошибки.

docs/ — изменения в документации.

style/ — изменения в стиле кода (форматирование, пробелы и т.д.).

refactor/ — рефакторинг кода без изменения функционала.

chore/ — задачи по обслуживанию проекта (обновления зависимостей, настройка окружения и т.д.).

### Пример

feat/user-authentication;

fix/login-page-error;

## Commit name pattern

<тип>[контекст]: <описание> [необязательное тело]

Feat - создание / добавление новой фичи / функции / класса / файла.

Update - обновление / дополнение фичи / функции / класса / файла.

Fix - исправлен уже обнаруженный баг / ошибка или исправлен код, который мог привести к ошибке в следствии ранее внесённых изменений.

VIP - "version in progress" используя этот тип у вас появляется возможность сделать коммит в любой момент в случае если вам нужно срочно прервать работу над текущей задачей и вы не хотели бы потерять изменения. Примеры: закончился рабочий день, вам нужен отдых, вас выдёргивают на другой проект или срочный fix. Нет ничего критичного даже в отправке этого коммита в удаленный репозиторий - вы сообщили другим разработчикам что вы еще в процессе реализации.

Refactor - рефакторинг без изменения ожидаемого результата. Например, вы просто переписали функцию в декларативном стиле при этом возвращаемое значение или суть работы функции, не изменились.

Docs - добавлена или изменена документация.

Style – изменения в стиле кода (форматирование, пробелы и т.д.). Да, форматирование кода – отдельный коммит, а работа над фичей, или fix – отдельный коммит.

Chore - задачи по обслуживанию проекта (обновления зависимостей, настройка окружения и т.д.).

## Инфраструктура

### Требования

- Docker Desktop
- Node.js 24 LTS
- pnpm

### Запуск если все шаги снизу пройдены

Поднимет контейнеры, запустит бек и фронт одной командой

```bash
pnpm run dev
```

### Запуск

```bash
cp docker/.env.example docker/.env

pnpm install

pnpm docker:up
```

### Остановить

```bash
pnpm docker:down
```

### Показать контейнеры

```bash
pnpm docker:ps
```

### Логи

```bash
pnpm docker:logs
```

### Удалить все контейнеры

```bash
pnpm docker:clean
```
